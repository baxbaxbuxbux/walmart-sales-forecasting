
# Walmart Recruiting - Store Sales Forecasting


მოცემული პროექტი ეხება Kaggle-ის ერთ-ერთ კონკურსს, სადაც მიზანია Walmart-ის გაყიდვების პროგნოზირება. მოცემული გვაქვს მონაცემთა ბაზა რომელიც სამადაა გახლეჩილი, train, stores და features ფაილებად.
პროექტზე მუშაობისას გავტესტეთ სხვადასხვა time series მოდელი, როგორებიცაა :
* **კლასიკური სტატისტიკური time-series მოდელები**: SARIMAX, Prophet
 * **Tree-Based მოდელები**:  XGBoost და LightGBM
 * **Deep Learning მოდელები**: N-beats, DLinear და TFT
 ქვევით დეტალურად განვიხილავთ თითოეულ მოდელსა და პროექტის ნაწილს


## EDA
მონაცემთა კვლევითი ანალიზე წარმოდგენილია `EDA.ipynb` ფაილში
* **ზოგადი მიმოხილვა**: თავიდან შევამოწმეთ მონაცემების თითოეული სვეტისათვის შესაძლო მნიშვნელობები, რითიც გავარკვიეთ, რომ გვაქვს 143 კვირის მონაცემები, რითიც უნდა დავაპროგნოზოთ 39 კვირის გაყიდვები. შესაბამისად ლოგიკურია, რომ საწვრთნელი დატასეტი იგივე პროპორციით გავხლიჩოთ საწვრთნელ და ვალიდაციის სეტებად, როგორი პროპორციაც აქვთ ჩვენს სრულ საწვრთნელ დატასეტსა და ტესტ სეტს, შესაბამისად ვალიდაციას ვაკეთებთ 31 კვირაზე.

* **საშუალო გაყიდვების შემოწმება**: თითოეულ (store, dept) წყვილის საშუალო გაყიდვების მატრიცის აგებით დავინახეთ ზოგადი სურათი, რომ ზოგ მაღაზიას ყველა დეპარტამენტში მეტი გაყიდვები აქვს ვიდრე სხვას და იგივენაირად ზოგიერთი დეპარტამენტი ყველა მაღაზიაში მეტად შემოსავლიანია ვიდრე სხვა, ასევე ჩანს, რომ ბევრი დეპარტამენტი ყველა მაღაზიაში არაა.
* **სრული გაყიდვების გრაფიკები**: დროის მიხედვით გაყიდვების გრაფიკების აგებით ცხადად გამოჩნდა გაყიდვების დინამიკა, რომელიც ყველა მაღაზიაში მსგავსი იყო. თვალითაც ჩანდა ძლიერი სეზონურობა და დასვენებების გავლენა რამაც ცხადყო, რომ კლასიკურ სტატისტიკური მოდელებიდან ARIMA და SARIMA კარგ შედეგს ვერ მოგვცემდნენ, რადგან ARIMA სეზონურობას არ ითვალისწინებს, ხოლო SARIMA ცვლადებს, როგორიცაა isHoliday.
* **ცვლადების გავლენა გაყიდვებზე**: გამოვთვალეთ კორელაცია და mutual information გაყიდვების სხვადასხვა ცვლადებთან. მნიშვნელოვანი კორელაცია მხოლოდ მაღაზიების მახასიათებლებთან მივიღეთ, ანუ აღსანიშნავი წრფივი დამოკიდებულებები feature-ებთან არ გვაქვს, თუმცა mutual information-ით (რომელიც არაწრფივ დამოკიდებულებებსაც ითვალისწინებს) აღმოჩნდა, რომ მცირე გავლენას ისეთი feature-ებიც ახდენდა, როგორებიცაა უმუშევრობა და CPI.
* **N/A**: ვნახეთ, რომ Markdown-ებში ინფორმაციის დიდი ნაწილი არ გვქონდა რაც მათ გამოყენებას ართულებს. ასევე ვნახეთ, რომ სხვადასხვა (store, dept) წყვილისათვის time-series სხვადასხვა სიგრძის იყო, მაგალითად გვიან იწყებოდა, რაც ნიშნავს, რომ ისეთი მწკრივების პროგნოზირება, რომლებშიც ძალიან მცირე მონაცემები გვაქვს ძალიან რთულია და შესაბამისად გვაქვს ფაქტობრივად აღმოუფხვრელი loss-ის წყარო.

# მოდელები

## კლასიკური სტატისტიკური მოდელები
კლასიკური სტატისტიკური მოდელები წარმოდგენილია `model_experiment_Classical.ipynb` ფაილში და დავ-track-ეთ [MLFlow-ზე](https://dagshub.com/grtsir22/walmart-recruiting.mlflow/#/experiments/1/runs?searchFilter=&orderByKey=attributes.start_time&orderByAsc=false&startTime=ALL&lifecycleFilter=Active&modelVersionFilter=All+Runs&datasetsFilter=W10%3D)


### SARIMAX
SARIMAX დროითი მწკრივების ანალიზის კლასიკური სტატისტიკური მოდელია. გარე(exogenous) feature-ებად მას მივაწოდეთ ['IsHoliday', 'Temperature', 'Fuel_Price', 'CPI', 'Unemployment']  მისი შედეგები 31 კვირიან ვალიდაციის მონაცემებზე იყო **4883.1762**, რაც საკმაოდ ცუდი შედეგია, რაც ასეთი სუსტი მოდელისთვის გასაკვირი არცაა, როგორც EDA-ს დროს ვნახეთ გაყიდვებს ფასებთან წრფივი კავშირები არ აქვთ, შესაბამისად SARIMAX ვერ ახერხებს ამ კავშირების დაჭერას.
### Prophet
Prophet-საც იგივე feature-ები გავატანეთ და შედეგი ვალიდაციაზე **3654.8721** იყო, რაც მნიშვნელოვანი გაუმჯობესებაა, თუმცა არა საკმარისი. ამის მიუხედავად გასათვალისწინებელია, რომ ტესტირებისას ერთადერთი ცვლადი რომელიც გვაქვს 'IsHoliday' არის, რაც ნიშნავს, რომ თუ დანარჩენი feature-ების გამოყენება გვსურს ისინი თავად უნდა დავაპროგნოზოთ, რაც ამ მოდელით ვერ მოგვცემს წესიერ შედეგს, ანუ თუ რეალურად გვსურს ტესტზე გაშვებადი მოდელის შექმნა მას მხოლოდ 'IsHoliday' უნდ გავატანოთ. ამის გაკეთების შემდეგ შედეგად ვიღებთ **3127.9375**, რაც კიდევ გაუმჯობესებაა, რაც სავარაუდოდ იმით უნდა იყოს გამოწვეული, რომ ზედმეტი feature-ები უბრალოდ ხმაურს ზრდის არაწრფივი კავშირების გამო.



## Deep learning
ეს მოდელები დავ-track-ეთ  : [WandB-ზე](https://wandb.ai/grtsir22-free-university-of-tbilisi-/walmart-sales-forecasting?nw=nwusergrtsir22)
### DLinear
ეს მოდელი წარმოდგენილია `model_experiment_DLinear.ipynb` ფაილში.
DLinear-ის საბაზისო მოდელისათვის გამოვიყენეთ შემდეგი კონფიგურაცია : "input_size": 52, "max_steps": 500, "learning_rate": 1e-3,"moving_avg_window":25,
რამაც შედეგი მოგვცა **2116.21339**, ამის შემდეგ დამაინტერესა input_size-ს გაორმაგება თუ გააუმჯობესებდა შედეგს მეტი კონტექსტის მიცემის შედეგად, თუმცა ამის მაგიერ სერიოზული გაუარესება გამოიწვია. ამან მიგვანიშნა იმაზე, რომ ეს ცვლადი სერიოზულ გავლენას ახდენდა შედეგზე, რის გამოც მისი ოპტიმიზაცია grid search-ით ვცადე, თუმცა აღმოჩნდა რომ ოპტიმალური მართლაც 52 ყოფილა, რაც მიგვანიშნებს იმაზე, რომ ძლიერი 1 წლიანი სეზონურობა გვაქვს, რაც ლოგიკურია, მაგრამ დამატებითი კონტექსტი მოდელს ზედმეტად კომპლექსურს ხდის და პერფორმანსს ვერ აუმჯობესებს. ამის შემდეგ grid search გავუშვით moving_avg_window-ზე, max_steps-ზე და learning_rate-ზე. თუმცა ვერცერთმა ამ მცდელობამ გაუმჯობესება ვერ გამოიწვია, ანუ საბოლოოდ საუკეთესო DLinear-მა loss **2116.21339** მოგვცა.

### N-beats
ეს მოდელი წარმოდგენილია `model_experiment_n_beats.ipynb` ფაილში.
N-beats-ის საბაზისო მოდელისათვის გამოვიყენეთ შემდეგი კონფიგურაცია : "input_size": 52, "max_steps": 500,"learning_rate": 1e-3, რამაც შედეგი მოგვცა **1632.79**, რაც DLinear-თან შედარებით რადიკალური გაუმჯობესებაა. ამის შემდეგ input_size-ს და ბლოკების კონფიგურაციის შეცვლა ვცადე [idnetity, indetity, identity]-ით, ესენიც grid search-ით ავარჩიე, input_size-ც, learning_rate-ც და max_steps-იც. საინტერესოა, რომ აქ input_size = 66-მა მოიტანა საკმაო გაუმჯობესება DLinear-ისგან განსხვავებით. ბლოკების კონფიგურაცია კომპლექსურობის გაზრდასთან ერთად დაახლოებით იგივე შედეგზე გვაბრუნებდა, რაც საწყისი ['identity', 'seasonality', 'trend'] საბოლოოდ ოპტიმალური კონფიგურაცია მივიღეთ : 
'input_size': 66, 'learning_rate': 0.002, 'seed': 42, 'max_steps': 2600, 'stack_types': ['identity', 'identity', 'identity'], 'n_blocks': [1, 1, 1], 'mlp_units': [[512, 512], [512, 512], [512, 512]]. ნაბიჯების და სიჩქარის გაზრდა მოდელს აუმჯობესებებდა, თუმცა მოდელის კომპლექსურობის გაზრდა პირიქით აუარესებდა, რადგან სავარაუდოდ overfit-ში გადადიოდა. ამ მოდელმა loss მოგვცა **1520.75**, რაც ჯერ-ჯერობით საუკეთესო შედეგია ჩვენი მოდელებისათვის, შესაბამისად ეს მოდელი გადავატრენინგეთ მთელ დატასეტზე 'horizon': 40-ით რათა ტესტ სეტზე მივიღოთ ჩვენი პროგნოზები.
#### feature-ების პროგნოზი
ბოლოს მიღებული n-beats მოდელი გამოვიყენე 'Unemployment'-ს და 'CPI'-ს დასაპროგნოზებლად რითიც loss-ები მივიღეთ  0.26  და 1.12 რაც შესაბამისი feature-ების მნიშვნელობებთან შედარებით საკმაოდ კარგი შედეგებია, ანუ ამ მოდელის პროგნოზები შეგვიძლია გამოვიყენოთ იმ მოდელებში რომლებიც მათ იყენებენ.
